## DER INTEGER encoding

A DER-encoded INTEGER is a TLV (Tag-Length-Value) triplet:

### Tag

The tag byte must be `0x02` (INTEGER, primitive form).

### Length

The length field encodes the number of value (content) bytes that follow:

- **Short form** (1 byte): If the length is 0–127, it is encoded as a single byte with bit 7 clear. For example, length 3 is encoded as `0x03`.
- **Long form** (2–3 bytes): If the length is 128–65535, the first byte is `0x80 | n` where `n` is the number of subsequent length bytes (1 or 2), followed by `n` bytes encoding the length in big-endian with no leading zeros. For example, length 128 is `0x81 0x80`; length 256 is `0x82 0x01 0x00`.
- The **indefinite form** (`0x80`) is not allowed in DER.
- Short form must be used when possible — long form is only valid for lengths that cannot be represented in short form (i.e., 128 and above).
- Long-form length bytes must not have leading zeros.
- This validator only needs to support lengths representable in at most 2 subsequent bytes (max length = 65535). Reject any length field claiming more than 2 subsequent bytes.

### Value (content)

The value bytes encode a signed integer in big-endian two's complement. The content must be at least 1 byte (length 0 is invalid).

The encoding must be minimal: the value must use the fewest possible bytes. Per X.690 §8.3.2, if the contents octets comprise more than one octet, then the bits of the first octet and bit 8 of the second octet shall not all be ones and shall not all be zero.

Examples:
- `02 02 00 80` is the valid encoding of 128. The leading `0x00` is required because `0x80` alone would be interpreted as -128 in two's complement.
- `02 01 7F` is the valid encoding of 127. No padding byte is needed because `0x7F` has bit 7 clear, so it is already positive.
